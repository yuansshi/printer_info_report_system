# 项目计划与改进路线图

## 1. 目标

把当前可运行的脚本升级为可配置、可测试、可自动运行、可审计的内部打印机报表服务，并逐步支持可靠的黑白、彩色、复印和扫描计数分析。

## 2. 实施原则

1. **先保证数据正确，再自动化。** 无法解释的计费器字段保持原始值。
2. **保留原始证据。** 原始邮件、标准化结果、运行清单和最终工作簿可以相互追溯。
3. **完整模式与兼容模式分离。** 不为逐字符兼容而静默丢失有效状态。
4. **幂等运行。** 同一日期重复运行应产生相同数据，不重复处理或修改邮箱。
5. **配置与密钥分离。** 路径、日期和业务模式可配置；密钥永不进入仓库。

## 3. 里程碑

### M0：项目基线和可重复运行（当前）

状态：**已完成**

交付物：

- 只读 IMAP 提取和邮件回放。
- 中英文及旧版 Fuji Xerox 模板解析。
- 单日三页签工作簿。
- 五日报表、设备汇总和分析页。
- 从 2026-06-01 开始、支持日/周/月统计的本地运营 Dashboard。
- 每日历史分区、修订版本、质量门禁和自动发布链路。
- 6 月 20 日参考数据、7 月 2 日和五日数据验证。
- 项目 README、当前状态、运行手册和路线图。

### M1：工程化基础

优先级：**P0，进行中**

已完成：Dashboard 配置文件、日期范围参数、`.gitignore` 和运行 manifest。Excel 报表入口仍需统一。

工作项：

- 建立标准 Python/Node 项目结构和统一 CLI。
- 将邮箱、路径、时区、模板和输出目录移到配置文件或命令行参数。
- 提取共享的映射、Excel 日期、去重和样式模块，消除两个 JavaScript 生成器中的重复代码。
- 支持 `--date`、`--from`、`--to` 和 `--days`，移除五日脚本中的固定日期。
- 加入 `.gitignore`，明确排除 `.env`、原始邮件、输出和缓存。
- 生成每次运行的 manifest：日期范围、UID 数、解析数、设备数、映射问题、代码版本和输出校验值。

验收标准：

- 新机器只需配置环境和模板路径即可运行。
- 同一日期重复运行的标准化 JSON 和状态行集合一致。
- 单日与多日生成器使用同一套核心规则。
- 命令以非零退出码报告缺失配置、解析失败和导出失败。

### M2：测试和数据质量门禁

优先级：**P0**

工作项：

- 建立匿名化邮件 fixture，覆盖中文、英文、旧版模板、HTML、缺失日期和异常编码。
- 为标签提取、章节分类、映射选择、日期转换和去重建立单元测试。
- 将 2026-06-20 的 52 条窗口数据和 11 条状态结果作为 golden test。
- 为 2026-07-02 建立 123 条通知、39 条状态的集成测试。
- 增加映射表质量门禁：空编号、重复编号、缺失客户、缺失位置、非文本编号和前导零风险。
- 定义解析率阈值；低于阈值时停止自动发布并生成待检查列表。

验收标准：

- CI 能在无邮箱凭据情况下使用 fixture 完成全部测试。
- 解析规则变更不会破坏已验证模板。
- 每份报表都有机器可读的验证摘要。

### M3：计费器语义和用量模型

优先级：**P1**

工作项：

- 收集各型号的计费器 `[1]-[5]` 官方定义、设备网页说明或厂商 MIB。
- 建立 `model + firmware -> counter mapping` 版本化字典。
- 区分累计计数和状态事件，按设备保存计数器快照。
- 计算相邻快照增量，并处理换机、清零、倒退和异常跳变。
- 输出黑白、彩色、复印、打印和扫描指标时同时保留原始计数器及映射版本。

验收标准：

- 每个已支持型号都有来源、样本和人工核对记录。
- 无映射型号继续展示原始计数器，不产生猜测值。
- 增量异常被标记而不是计入正常使用量。

### M4：映射表治理

优先级：**P1**

工作项：

- 将机身编号统一存储为文本并设置唯一约束。
- 建立客户名称规范化和别名表。
- 提供“新增设备、缺失位置、重复编号、冲突客户”的待办清单。
- 为映射变更记录修改人、时间和原因。
- 报表运行前执行映射预检，严重冲突时阻止发布。

验收标准：

- 一个有效机身编号只有一个当前映射。
- 前导零和非纯数字编号不会丢失。
- 所有自动生成报表都能追溯到映射表版本。

### M5：可靠性和无人值守运行

优先级：**P1，部分完成**

已完成：Dashboard 的 IMAP 重试、按日幂等分区、最近 3 天重抓、运行锁、远程 systemd timer 示例、原子发布和上一成功版本保护。UID 检查点、结构化告警和 Excel 自动发布仍待完成。

工作项：

- 增加 IMAP 重试、指数退避、超时分类和连接恢复。
- 保存 UID 检查点，支持增量提取和补跑。
- 增加运行锁，防止同一日期并发生成。
- 使用结构化日志记录阶段、耗时、计数和错误类别。
- 使用 systemd timer、cron 或容器 CronJob 每天自动运行。
- 生成 Dashboard 数据快照并原子替换，保留最后一次成功版本。
- 失败时发送简要告警，成功时发布工作簿位置和质量摘要。

验收标准：

- 网络短暂失败可以自动恢复。
- 重复调度不会产生重复数据或损坏文件。
- 运维人员可以仅根据日志判断失败阶段和重跑范围。

### M6：监控和直接设备数据

优先级：**P2**

工作项：

- 将邮件数据与 SNMP Printer-MIB、Host Resources MIB 和厂商 MIB 数据关联。
- 记录设备在线状态、耗材百分比、页数计数和告警状态。
- 建立历史数据库和轻量监控界面。
- 将静态 Dashboard 升级为受控的数据 API 和可配置刷新机制。
- 比较邮件、SNMP 和设备网页数据，识别缺报或延迟。

验收标准：

- 邮件报表仍可独立运行。
- SNMP 不可达不会阻止邮件报表发布。
- 所有跨来源指标有明确优先级和冲突处理规则。

## 5. 建议的下一迭代

建议下一迭代只完成以下五项，避免同时扩张业务范围：

1. 建立统一配置和 CLI，移除硬编码日期与路径。
2. 把 6 月 20 日、7 月 2 日和旧版 `机身编号` 模板做成匿名化回归 fixture。
3. 提取共享映射和去重模块，使单日与多日报表使用相同实现。
4. 增加映射表预检报告，并将重复编号和缺失位置作为显式质量门禁。
5. 建立计费器型号调查表，只记录有来源的字段解释，不立即开发用量结算。

## 6. 报表功能的完成定义

一个报表功能只有在以下条件全部满足时才算完成：

- 有明确日期口径、输入数据和字段定义。
- 有匿名化测试样本和自动化测试。
- 同一输入重复运行得到一致结果。
- 未解析邮件、未知设备和映射问题不会被静默忽略。
- Excel 数值、公式、样式和页签均通过程序检查与视觉检查。
- 运行日志不包含密码、授权码或完整敏感邮件正文。
- 文档、运行手册和项目计划同步更新。
